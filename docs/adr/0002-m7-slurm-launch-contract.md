# ADR 0002: Stage Slurm support behind an unreachable fixed contract

- **Status:** Accepted
- **Date:** 2026-07-19
- **Decision owners:** easy_pipe maintainers
- **Scope:** M7.0 Slurm/HPC execution support

## Context

The current production execution path is intentionally limited to the
Nextflow local executor. Although `PipelineSpec` and `ExecutionPlan` can record
`slurm` and the compiler emits a reviewed placeholder configuration, the
execution profile, approval gate, remote preflight, and Remote Executor do not
authorize Slurm execution.

M7 must add a scheduler without turning either the Controller or Remote
Executor into a general-purpose command service. It must also preserve the
existing bindings among generated artifacts, the execution profile, fresh
preflight evidence, explicit actor approval, HMAC authentication, deployment,
run state, status, and resume.

The frozen public JSON Schema v1 catalog and Remote Executor protocol v1 have
exact-field contracts. Adding scheduler fields or operations to those v1
contracts in place would silently broaden their meaning.

## Decision

### 1. Begin with unreachable pure scheduler primitives

M7.0a introduces only dependency-free Slurm contract primitives and their
offline tests in the Remote Executor package. These primitives may validate a
strict scheduler policy, construct fixed argument vectors, parse bounded
scheduler output, and map recognized scheduler states to internal states.
Its reconciliation primitives encode rules only; they do not perform scheduler
discovery or mutate scheduler or run state.

They are deliberately unreachable from the Remote Executor dispatcher. M7.0a
does not import them into the current configuration, preflight, submit, status,
resume, or service entry points. It does not invoke a subprocess, contact a
scheduler, create scheduler state, or make a Slurm project executable. Passing
the M7.0a tests is contract evidence only, not execution or cluster evidence.

This isolation lets the injection boundary and state grammar be reviewed
before any scheduler mutation becomes possible.

### 2. Fix the first supported topology

The first executable Slurm topology will be:

```text
Controller
    -> fixed Remote Executor on a login or service node
    -> one reviewed sbatch allocation
    -> Slurm compute node
    -> Apptainer with hash-pinned local SIF files
    -> Nextflow local task orchestration inside that allocation
```

It requires a shared filesystem for the immutable inputs, deployment, pinned
Nextflow JAR, Apptainer SIF files, work directory, output directory, container
cache, and the bounded scheduler artifacts required by the run.

M7.0 does not support Docker on Slurm, task-per-job Nextflow Slurm execution,
job arrays, heterogeneous jobs, federated or multi-cluster submission, another
scheduler, automatic data staging, environment modules, or arbitrary site
prologues. The first topology is single-cluster, so a cluster suffix is never
accepted as part of a job identifier, even if a deployed Slurm version can
emit one.

### 3. Keep scheduler input structured and closed

The future scheduler policy will admit only named, bounded fields such as
partition, optional account and QoS, a canonical time limit, submit timeout,
status polling interval, and maximum pending duration. Unknown fields are
rejected. There is no `extra_flags`, free-form argument, shell fragment,
environment overlay, `--wrap`, or user-supplied job script.

Scheduler commands are built by the Remote Executor as fixed argument arrays
for reviewed absolute executables. A submitted job identifier must be parsed
from one bounded canonical `sbatch --parsable` response and must identify one
ordinary numeric job. Array, step, cluster-suffixed, ambiguous, empty, or
multi-line identifiers fail closed. Each submission also carries one
hash-bound 64-character lowercase hexadecimal marker as its exact scheduler
job name. The marker is generated by the trusted execution path and is not a
user-selectable name or scheduler argument.

The batch script is rendered as one bounded byte string from a fixed template.
Its exact bytes are hashed and bound before scheduler mutation, then those same
bytes are supplied directly to `sbatch` on standard input. The implementation
must not write a script and later reopen it by pathname, because that would
separate the authenticated bytes from the submitted bytes. Changing any
execution-relevant scheduler value therefore requires new preflight evidence
and approval.

### 4. Preserve every frozen v1 boundary

M7.0a does not change:

- the frozen public JSON Schema v1 files or catalog;
- `ExecutionProfile` v1, whose runtime remains local-only;
- the CLI contract version or existing command behavior;
- Remote Executor protocol v1 operations, request fields, response fields, or
  HMAC canonical envelope;
- local-executor preflight, submission, status, reconciliation, or resume; or
- the registry, fixed FASTQ-QC graph, container identities, or generated
  project bytes.

A later PR must introduce explicitly versioned scheduler-aware profile,
Remote Executor configuration, and protocol contracts. It must not reinterpret
an existing v1 artifact. The exact scheduler policy must be included in the
execution-profile identity so that existing artifact, approval, signature, and
resume bindings cover it.

### 5. Require an actual compute-node preflight

Login-node readability is not compute-node evidence. Before a Slurm preflight
can pass or mint a one-use execution capability, a real bounded Slurm
preflight job must verify from a compute node:

- input readability and the reviewed path mapping;
- deployment and pinned Nextflow JAR visibility and identity;
- Apptainer executable and SIF visibility and identity;
- private work and output reservation behavior;
- cache visibility and writability;
- required free space; and
- the network-disabled runtime policy.

The result must bind the exact scheduler policy, profile hash, project hash,
input-set hash, paths, container hashes, and preflight identity. A queued,
timed-out, missing, malformed, or partially observed preflight is not a passed
preflight.

### 6. Reconcile scheduler state without guessing

Executable M7.0 support will use a fixed `squeue` query for active scheduler
state and a fixed `sacct` query for accounting and terminal evidence. Their
outputs are separate evidence sources: disappearance from `squeue` alone is
not evidence of success, failure, or absence. Every accepted observation must
bind the exact 64-hex job-name marker and the expected scheduler `Submit`
timestamp as well as the job ID. A record with a mismatched or missing marker
or submit time is not evidence for the run.

Initial submission uses a held job so an ambiguous or lost `sbatch` response
cannot allow workload execution before identity reconciliation. Lost-response
discovery is positive-only: exactly one scheduler record with the expected
marker and `Submit` binding may establish the job ID. Zero matches leave the
outcome unknown; they do not authorize resubmission. More than one match is a
state conflict. A fixed `sacct --duplicates` query is interpreted just as
strictly: multiple candidate records block reconciliation rather than being
selected by recency or another heuristic.

Only reviewed Slurm states are mapped to the internal submitted, running,
succeeded, or failed states. Unknown states and malformed or conflicting
records block reconciliation. Success requires a recognized completed state
and a successful exit status. Terminal scheduler state, exit evidence, exact
job ID, job-script hash, command hash, environment hash, profile hash, project
hash, bundle hash, and authorization identity must remain bound in durable
state and audit events.

Requeue or scheduler restart observations are indeterminate in M7.0. They must
not be normalized to submitted, running, succeeded, or failed, and require
explicit blocking reconciliation. This avoids treating a changed scheduler
attempt or lifecycle as the originally authenticated execution.

The run ID is durably reserved before scheduler mutation. A timeout, crash, or
lost `sbatch` response must enter reconciliation and must never trigger blind
resubmission. Duplicate-submit recovery must resolve the exact reserved run and
job identity or remain blocked.

### 7. Do not add cancellation in M7.0

M7.0 provides no `scancel` operation, generic signal interface, or arbitrary
scheduler command. Cancellation is deferred to a separate M7.1 decision. Any
future cancellation capability must bind an exact locally recorded job ID and
profile, require an explicit actor and approval, authenticate the complete
payload, audit before and after, and be idempotent.

### 8. Separate implementation evidence from real-cluster acceptance

Unit and integration fixtures may prove deterministic parsing, command
construction, fail-closed transitions, and response-loss recovery. They may
not be described as a real Slurm run. M7 is not complete until an operator has
retained evidence from a supported cluster showing a full synthetic-data run,
compute-node path failure detection, scheduler lifecycle reconciliation,
Apptainer and Nextflow identity rechecks, and unchanged local-executor
behavior.

## Delivery sequence

The scheduler path will be connected through separate reviewable changes:

1. **M7.0a:** unreachable pure Slurm policy, argument, identifier, output, and
   reconciliation-rule primitives with offline tests;
2. **M7.0b versioned contracts:** scheduler-aware execution profile, Remote
   Executor configuration, and remote protocol without changing frozen v1;
3. **M7.0c compute-node preflight contract:** dormant manifest, fixed template,
   polling transitions, evidence binding, timeout, and capability-gate rules;
4. **compute-node preflight activation:** trusted config loading, bounded stdin
   submission, durable scheduler-preflight state, a reviewed compute worker,
   and one-use capability issuance;
5. **submission and status:** fixed batch script and `sbatch`, exact job-ID
   persistence, held positive-only lost-response discovery, marker-and-submit
   binding, `squeue`/`sacct` reconciliation, terminal audit, and compatible
   resume;
6. **failure recovery:** lost-response, duplicate-submit, accounting-delay,
   executor-restart, and state-conflict tests;
7. **real-cluster acceptance:** operator-run synthetic evidence on the
   supported Apptainer/shared-filesystem topology; and
8. **M7.1:** separately designed, authenticated exact-job cancellation, if it
   is approved at all.

Each connection step must retain the existing local-executor regression suite
and keep Slurm fail-closed until all prerequisites for that step are present.

## Consequences

### Positive

- Scheduler parsing can be security-reviewed without exposing a mutation path.
- The first HPC topology has one clear container, filesystem, allocation, and
  orchestration model.
- Scheduler policy will inherit the existing artifact, preflight, approval,
  signature, state, audit, and resume bindings.
- Existing v1 users and local execution keep their exact behavior while the
  new contracts are versioned explicitly.

### Costs and trade-offs

- M7 requires staged contract and protocol evolution before a Slurm job can be
  launched.
- Sites that depend on modules, arbitrary `sbatch` flags, Docker, federations,
  job arrays, or non-shared storage are outside the first supported topology.
- Scheduler accounting can lag queue state, so terminal reconciliation may
  remain pending rather than guessing.
- Zero scheduler matches, duplicate accounting records, requeue, and restart
  observations deliberately remain blocked or indeterminate.
- A single allocation with Nextflow local orchestration is less flexible than
  native task-per-job scheduling, but it provides a smaller initial trust and
  recovery boundary.

## Rejected alternatives

### Extending frozen v1 contracts in place

Rejected because adding scheduler fields and meanings to exact frozen schemas
and protocol payloads would be an undocumented compatibility break.

### Generating scheduler commands on the Controller

Rejected because the Controller must not become an arbitrary remote argv or
shell generator. The Remote Executor owns the fixed scheduler command grammar
and validates all structured inputs again at the mutation boundary.

### Treating login-node checks as compute-node preflight

Rejected because login and compute nodes may have different mounts, runtime
paths, permissions, caches, and network policy.

### Retrying `sbatch` after an ambiguous response

Rejected because the first request may already have created a job. Recovery
must reconcile the exact durable run identity instead of risking a duplicate
real-data execution.

### Adding a generic cancel command in M7.0

Rejected because scheduler cancellation is a separate authenticated mutation
with actor, job-binding, idempotency, and audit requirements.

## Authoritative Slurm references

The executable implementation and its tests must be checked against the
official SchedMD documentation for the deployed Slurm version:

- [`sbatch`](https://slurm.schedmd.com/sbatch.html): batch submission and
  parsable job identifiers;
- [`squeue`](https://slurm.schedmd.com/squeue.html): jobs currently represented
  in the scheduler queue;
- [`sacct`](https://slurm.schedmd.com/sacct.html): accounting, job/step records,
  state, and exit-code fields;
- [job state codes](https://slurm.schedmd.com/job_state_codes.html): base states,
  flags, and their user-facing representations; and
- [job exit codes](https://slurm.schedmd.com/job_exit_code.html): the
  exit-code-and-signal representation used by Slurm accounting.

These references define scheduler behavior; they do not relax easy_pipe's
stricter allowlists, output limits, identity bindings, or fail-closed handling.
