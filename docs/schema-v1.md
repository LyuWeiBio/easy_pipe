# Frozen MVP schema v1

The MVP machine contract is schema version `1.0`. This version is independent
of the controller package version (`0.1.0`) and registry version (`1.0.0`).

## Catalog

The controller exposes a deterministic JSON Schema draft 2020-12 catalog:

```bash
biopipe schema list --json
biopipe schema show DatasetManifest --json
biopipe schema export --output build/schema-v1 --dry-run --json
biopipe schema export --output build/schema-v1 --json
```

Each document has:

- a stable filename and `$id` below
  `https://github.com/LyuWeiBio/easy_pipe/schemas/v1/`;
- `$schema: https://json-schema.org/draft/2020-12/schema`;
- `x-biopipe-schema-version: "1.0"`; and
- a canonical SHA-256 listed in `catalog.json`.

The aggregate `catalog_sha256` covers all 21 schemas in stable name order.
Canonical JSON files and `catalog.json` are installed as package data;
`schema list`, `schema show`, and `schema export` read those frozen bytes instead
of allowing the installed Pydantic version to redefine the public contract.
Exported JSON is deterministic and contains no build timestamps or local paths.

## Included contracts

The v1 catalog includes:

- `SourceProfile`, `ProbeRequest`, and `ProbeResponse`;
- `DatasetManifest`, `ManifestOverrides`, and `OverrideDiff`;
- `PipelineSpec`, `ExecutionPlan`, and `SoftwareLock`;
- `RegistryDocument`;
- `ExecutionProfile`, `PreflightReport`, `RunAuthorization`, `RunReport`,
  `StatusReport`, and `ReconciliationReport`;
- `ValidationReport`, `WorkflowTestReport`, `ValidationCommandReport`, and
  `TestCommandReport`; and
- `AuditEvent`.

CLI command envelopes such as version/catalog/dry-run summaries and the
controller's stable error envelope are governed by
`cli_contract_version: "1.0"`. Remote JSONL request/response framing is
governed separately by `protocol_version: "1.0"`. A matching number does not
make these contracts interchangeable.

## Strictness and integrity

Public models reject unknown fields and invalid cross-field combinations.
Readers reject duplicate JSON/YAML keys and non-finite JSON values where those
formats are accepted. Artifact-specific versions such as `manifest_version`,
`spec_version`, `plan_version`, `profile_version`, `report_version`, and audit
`schema_version` remain literal `"1.0"`.

A JSON Schema validates structure, not the complete security contract. Runtime
code also verifies embedded/canonical hashes, path relationships, registry
consistency, generated bytes, report status, freshness, filesystem identities,
and approval bindings. Schema success alone is never approval to execute.

## Compatibility policy

For frozen v1:

- field names, types, required/optional status, enum values, nesting, and
  additional-field behavior are part of the contract;
- changing canonical schema output changes the per-schema and catalog hashes
  and requires explicit review;
- existing meanings must not be weakened or silently broadened;
- new error codes may be added, but consumers must continue to treat unknown
  non-success codes as blocking;
- incompatible artifact changes require a new schema/protocol version and a
  documented migration; and
- the MVP provides no automatic migration or downgrade path.

Release CI compares current model generation, installed canonical resources,
and the complete committed v1 fixture set byte-for-byte. An intentional schema
change must update version policy, package resources, fixtures, documentation,
compatibility tests, probe/executor/controller readers, and release notes in one
reviewed change.

## Consumer guidance

1. Pin both the application release and the expected schema/catalog digest.
2. Validate with the exported schema, then apply the artifact-specific integrity
   and cross-field checks documented for the command.
3. Reject unsupported versions, duplicate keys, non-finite values, and unknown
   fields rather than stripping them.
4. Preserve the original artifact bytes and recorded SHA-256 for audit.
5. Never transform a sanitized manifest into executable input or interpret a
   passed validation report as real-data approval.

See the [CLI reference](cli-reference.md) for schema commands and the
[release checklist](release-checklist.md) for freeze verification.
